**1. DAE 当前架构边界**
- DAE 是 Spring Cloud，但具体组件是什么？Spring Cloud Gateway / Zuul / Nacos / Eureka / Config Server / Feign / Ribbon / OpenFeign / Resilience4j 哪些在用？

- 现在 DAE 前端技术栈是什么？React、Angular、Vue，还是 server-side template？

**2. 当前 Lead Validation 执行链**
- 现在线索进入 DAE 后，SQS / batch / manual create 三个入口分别落在哪些服务、哪些类/接口？
- 三个入口是否已经共用同一套 validation service，还是各自散落硬编码？
- 当前 validation 和 allocation 的边界在哪里？validation 失败后写 exception 的逻辑在哪个服务？
- Hard Stop、SKIPPED、DUPLICATE 这些状态现在是否已有统一枚举/表字段？

**3. 本期 Simulation 的真实范围**
- 业务方说做到第 3 阶段 simulation，是指：
  - 只做沙箱手动输入 Lead 模拟？
  - 还是支持历史生产 Lead 回放？
  - 是否需要 before/after 影响分析？
  - 是否允许保存一个“拟调整配置”但不发布？
- Simulation 是否需要调用真实后端 validation logic，还是允许使用 mock/snapshot logic？
- 模拟结果是否要求与生产 validation 100% 一致？如果不一致，容忍边界是什么？

**4. 配置系统的权限和发布边界**
- 本期是否完全不允许业务修改生产规则？
- 是否允许业务创建 draft configuration 用于 simulation？
- 谁能看规则、谁能模拟、谁能保存 draft、谁能提交变更请求？
- DAE 当前权限体系是什么？JWT、SSO、OneConnect role、Spring Security、还是网关统一鉴权？
- 是否已有 `DAE_RULE_ADMIN` 或类似角色？

**5. 规则数据来源**
- 当前规则清单是来自代码扫描、DB 表、URule repository，还是手工维护的 registry？
- 是否已有 `validation_rule_registry` 表？如果没有，本期是否允许新增？
- 每条规则是否已有稳定的 `rule_code`、`rule_name`、`execution_order`、`hard_stop`、`dependency_condition`？
- 规则说明给业务看的文本从哪里维护？代码注解、DB、配置文件，还是新表？

**6. Simulation 输入数据**
- 模拟 Lead 的 payload schema 是哪个版本？是否已有 Standard Payload？
- 是否可以使用脱敏历史 Lead 做回放？
- 历史 Lead 数据从哪里取？DAE DB、Snowflake、S3、audit table？
- PII/POPIA 合规要求是什么？手机号、邮箱、ID number 在 simulation UI 能否展示？
- 是否需要 preset 场景：valid lead、missing campaign、duplicate、deceased、PSI invalid 等？

**7. Simulation 执行方式**
- 如果调用真实 validation service，是否允许 simulation 走只读模式，避免写入 exception、触发 allocation、发送消息？
- 是否需要一个 `simulation=true` context flag，让 validation 只返回结果不产生副作用？
- 现有 validation 代码是否容易做到 dry-run？还是很多逻辑会直接写库/发消息？
- 外部依赖如 InteGreat、customer service、adviser service 在 simulation 中调用真实接口、mock，还是读取快照？

**8. 结果解释和审计**
